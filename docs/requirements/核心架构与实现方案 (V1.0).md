# 🔮 沉浸式塔罗占卜 Web 应用 - 核心架构与实现方案 (V1.0)

## 1. 数据结构设计 (Data Structure Design)

在前端进行状态管理时，我们需要建立清晰的领域模型。以下是核心数据实体的结构定义：

### 1.1 卡牌实体 (Card Entity)

用于定义单张塔罗牌的基础元数据。

- **id**: 唯一标识符（例如：`maj-00` 代表愚者）。
- **name**: 卡牌名称（例如：The Fool）。
- **suit**: 牌组（大阿尔卡纳、权杖、圣杯、宝剑、星币）。
- **imageUrl**: 牌面高清图的静态资源路径。
- **keywords**: 核心关键词数组（用于本地字典库占卜）。
- **element**: 元素属性（火/土/风/水，用于辅助 AI 生成学术派解读）。

### 1.2 牌阵实体 (Spread Entity)

用于定义占卜的几何阵型与位置含义。

- **id**: 唯一标识符（例如：`holy-triangle`）。
- **name**: 牌阵名称（圣三角牌阵）。
- **cardCount**: 需要抽取的卡牌总数。
- **positions**: 位置数组。每个位置包含：
  - `index`: 抽取顺序。
  - `meaning`: 该位置的代表含义（例如：“过去的业力”、“当前的能量”、“未来的走向”）。
  - `coordinates`: 2.5D 视图下的相对坐标点（X, Y），用于动画归位。

### 1.3 占卜会话状态 (Reading Session State)

贯穿整个单页面应用生命周期的核心状态树。

- **step**: 当前阶段（枚举值：`SETUP` / `SHUFFLE` / `DRAW` / `REVEAL` / `INTERPRET`）。
- **userQuestion**: 用户输入的占卜意图/问题。
- **selectedSpread**: 当前选中的牌阵实体。
- **drawnCards**: 用户抽中的卡牌数组。每项包含：
  - `cardId`: 关联的卡牌 ID。
  - `isReversed`: 布尔值，是否为逆位。
  - `positionIndex`: 对应牌阵中的位置索引。
- **interpretationText**: 最终生成的解读长文本。

### 1.4 用户配置 (User Settings)

存储于本地 `LocalStorage`。

- **apiKey**: 用户填写的 LLM API 密钥（加密或明文混淆存储）。
- **interactionMode**: 交互偏好（枚举值：`GESTURE` 手势 / `TOUCH` 键鼠）。
- **audioEnabled**: 布尔值，环境音与音效开关。

## 2. 主要页面/组件及其职责 (Component Architecture)

项目采用基于 React/Vue 的单页面组件化架构，按职责划分为容器层、视图层和通用 UI 组件层。

### 2.1 容器层 (Container)

- **`AppContainer` (主控台)**：
  - 全局状态的提供者（Provider）。
  - 统筹各视图层的按需加载与平滑切换。
  - 全局音频控制器（Howler.js 实例挂载点）。

### 2.2 视图层 (Views - 对应业务阶段)

- **`PortalView` (入口与契约视图)**：
  - **职责**：渲染初始羊皮纸界面，处理“预授权仪式弹窗”，获取摄像头权限。收集用户问题与牌阵选择，提供 API Key 设置入口。
- **`RitualSpaceView` (核心仪式/交互视图)**：
  - **职责**：承载最核心的 2.5D 交互逻辑。
  - **子组件 `CameraTracker`**：后台运行 MediaPipe，处理视频流，提取手部特征点坐标，将坐标系归一化并传递给主控。
  - **子组件 `CardDeckManager`**：管理 78 张牌的 CSS3D 堆叠。监听手势坐标，执行洗牌（圆周运动检测）和抽牌（捏合动作检测）的视觉反馈。
- **`RevelationView` (启示与解读视图)**：
  - **职责**：展示最终牌阵，翻转卡牌。处理 AI 接口请求，流式展示（打字机效果）学术派解读结果。
  - **子组件 `ExportEngine`**：利用 `html2canvas` 将当前视图的结果区域渲染为带有火漆印章的长图并触发下载。

## 3. 核心流程的实现思路 (Core Flow Logic)

### 3.1 手势捕捉与意图判定机制

1. **坐标映射**：将 MediaPipe 返回的食指尖（Landmark 8）坐标，按比例映射到浏览器视口的 X/Y 像素坐标，驱动屏幕上的“金色光球”移动。
2. **洗牌判定 (圆周检测)**：记录光球近 1 秒内的轨迹坐标点。通过计算这些点组成的向量角度变化，若累计旋转度数超过阈值（如 $1800^\circ$，即 5 圈），则触发“洗牌完成”事件。
3. **抽牌判定 (捏合检测)**：计算大拇指尖（Landmark 4）与食指尖（Landmark 8）的欧氏距离。当距离小于设定阈值（如屏幕宽度的 3%），且光球停留在某张待抽卡牌的 DOM 区域上时，判定为“选中”。

### 3.2 动态 Prompt 组装与 AI 解读

在用户完成抽牌，切换至 `RevelationView` 时触发：

1. **组装上下文**：系统读取 `Reading Session State`。
2. **构建 System Prompt**：注入“专业神秘学学者”的人设限制，要求输出必须包含符号学、元素对应，并保持学术严谨的口吻。
3. **构建 User Prompt**：组合格式为 `[用户问题] + [牌阵名称] + [位置1: 牌名+正逆位] + [位置2: 牌名+正逆位]...`。
4. **请求与渲染**：通过前端直接调用 LLM API，通过 SSE (Server-Sent Events) 接收流式响应，驱动打字机组件逐字渲染。若未配置 API Key，则按照组合逻辑从本地字典库拼接默认文案。

### 3.3 命运卷轴生成机制

1. **DOM 准备**：在页面底层隐藏渲染一个完整的、排版精美的“结果海报” DOM 节点（包含羊皮纸背景图、牌阵截图、完整解读文本、时间戳和火漆印章）。
2. **画布转换**：用户点击下载时，触发 `html2canvas` 遍历该隐藏节点，计算所有 CSS 样式，将其绘制到内存中的 Canvas 上。
3. **导出下载**：将 Canvas 转换为 Base64 图片数据流（PNG 或 JPEG），创建虚拟 `<a>` 标签触发浏览器本地下载。

## 4. 可能遇到的技术难点 (Technical Challenges & Mitigations)

### 4.1 性能瓶颈：MediaPipe 与动画争抢主线程资源

- **风险**：MediaPipe 每秒需处理 30 帧以上的画面，如果直接在 JS 主线程运行，会导致 Framer Motion 的卡牌动画掉帧、卡顿，破坏沉浸感。
- **预案**：必须将 MediaPipe 的图像处理和特征点计算逻辑剥离到 **Web Worker** 中运行。主线程只负责接收 Worker 传回的轻量级坐标数据并渲染动画。

### 4.2 光线环境对识别的致命影响

- **风险**：纯视觉捕捉在用户逆光、暗光环境下识别率极低，会导致手势抽牌完全失效。
- **预案**：设计**智能降级机制**。实时监测 MediaPipe 返回的置信度得分（Confidence Score）。若连续 5 秒置信度低于 0.5，界面需优雅地淡出提示：“感知场受到干扰，已自动开启触屏/鼠标共鸣模式”，无缝切换到备用交互。

### 4.3 `html2canvas` 对 CSS3D 的渲染缺陷

- **风险**：这是企业级前端常踩的坑。`html2canvas` 无法完美识别和渲染带有 `perspective`、`rotateX/Y` 的复杂 2.5D 变形。如果在结果页直接截图翻转后的 3D 牌面，可能会出现图像扭曲或白屏。
- **预案**：生成长图时，**不可直接截取交互视图的 DOM**。必须在导出专用的隐藏 DOM 中，将塔罗牌替换为**纯平面的 2D 静态图片**，辅以简单的 CSS 阴影来伪造立体感，以此确保导出的截图万无一失。

### 4.4 AI 输出的格式失控

- **风险**：大语言模型可能会无视 Prompt 指令，输出不带段落、包含 Markdown 乱码的文本，或者在开头加上“好的，这是您的解读”等破坏氛围的废话。
- **预案**：
  - 在 System Prompt 中使用强指令（如：`DO NOT output any conversational filler. Output strictly in the specified JSON/Markdown structure.`）。
  - 在前端编写一个防抖的文本解析器，过滤掉非学术风格的语气词和不合规的标签后再渲染到页面上。
