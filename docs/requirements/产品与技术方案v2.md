# 📄 沉浸式手势交互塔罗占卜 Web 应用 - 系统设计与实现方案 (V1.0 MVP)

## 1. 文档概述 (Document Overview)

- **项目背景：** 打造一款基于 Web 的沉浸式塔罗占卜应用，以“手势交互”为核心卖点，辅以复古羊皮纸视觉、白噪音沉浸感以及专业的神秘学 AI 解读。
- **文档目的：** 明确 MVP 期（2.5D 方案）的技术架构、模块划分、数据流向及容灾策略，作为研发团队的开发依据。
- **架构选型基调：** 纯前端架构（Serverless 理念），采用 SPA（单页面应用）模式，重客户端渲染与计算。

## 2. 系统整体架构设计 (System Architecture)

系统采用**分层组件化架构**，主要划分为以下四个层次：

1. **基础设施层 (Infrastructure)：** 浏览器 API（Camera/Web Audio/LocalStorage）。
2. **核心引擎层 (Core Engines)：** MediaPipe 视觉引擎（运行于 Web Worker 以防主线程阻塞）、Howler.js 音频引擎、牌阵与随机数生成算法库。
3. **业务逻辑层 (Business Logic)：** 状态机控制器（State Machine）、手势意图解析器、双轨解牌控制器（本地字典/LLM）。
4. **表现层 (Presentation)：** 基于 React + Framer Motion 的 2.5D 渲染界面、CSS Shader 滤镜、html2canvas 导出模块。

## 3. 核心业务模块划分 (Core Modules)

### 3.1 预授权与配置模块 (Setup & Config)

- **仪式感授权：** 拦截浏览器默认行为，先弹出“灵界链接”契约 UI，用户点击同意后再触发 `navigator.mediaDevices.getUserMedia` 请求摄像头。
- **环境探测与降级：** 探测摄像头是否可用。若拒绝或无设备，自动降级激活“实体触控（鼠标/触摸）”模式，并在界面点亮对应的降级状态指示灯。
- **本地存储：** 利用 `LocalStorage` 安全缓存用户的 API Key、音量设置及偏好牌阵，实现“阅后即焚”或本地持久化。

### 3.2 机器视觉与手势解析模块 (CV & Interaction)

- **流处理引擎：** 引入 `@mediapipe/hands`。为保证性能，抽帧处理摄像头视频流。
- **意图识别器 (Intent Recognizer)：**
  - *光标映射：* 提取食指指尖（Landmark 8）坐标，应用平滑算法（如卡尔曼滤波防抖）后映射为屏幕上的金色光球坐标。
  - *洗牌判定：* 记录指尖轨迹，计算圆心角积分。当积分达到 $5 \times 360^\circ$ （即 5 圈）时，抛出 `SHUFFLE_COMPLETE` 事件，触发自动收牌逻辑。
  - *抽牌判定：* 计算拇指指尖（Landmark 4）与食指指尖（Landmark 8）的欧几里得距离 $D$。当 $D < \text{阈值}$ 且持续 $100\text{ms}$ 以上，抛出 `PINCH_SELECT` 事件。

### 3.3 占卜逻辑与牌阵引擎 (Divination Engine)

- **牌库定义：** 定义 78 张牌的数据结构 `{ id, name, arcana_type, element, astrological_association }`。
- **洗牌算法：** 采用标准的 Fisher-Yates 洗牌算法，确保随机分布的密码学安全级别（避免伪随机偏差）。
- **正逆位生成：** 抽卡确认瞬间，通过 `Math.random() > 0.5` 赋予该卡牌 `isReversed` 状态。
- **牌阵插槽调度 (Spread Scheduler)：** 定义不同牌阵（如圣三角）的几何坐标系 $(X, Y, \text{Rotation})$，抽卡后将实体卡牌通过动画引擎推送到对应插槽。

### 3.4 释义与 AI 大模型调度模块 (Interpretation & LLM)

- **双轨路由：** 检查配置中是否存在有效的 API Key。若无，路由至本地 156 词条 JSON 字典。若有，触发大模型 API。
- **System Prompt 工程 (核心资产)：**
  - 设定 Persona：精通赫尔墨斯主义、卡巴拉生命树的学术派塔罗师。
  - 输入组装：`{用户问题} + {牌阵定义} + {抽中牌组及正逆位}`。
  - 输出规范要求大模型按四个维度结构化返回：`【元素与星象】`、`【符号学解析】`、`【深层意涵】`、`【启示建议】`。
- **流式接收：** 采用 Server-Sent Events (SSE) 处理大模型响应，前端实现“打字机”效果，缓解等待焦虑。

### 3.5 渲染与视听引擎 (Rendering & Audio)

- **2.5D 物理表现：** 使用 CSS `perspective` 构建 3D 舞台，通过 Framer Motion 控制卡牌的 `rotateY` (翻转) 和 `translateX/Y` (飞入)，实现伪 3D 沉浸感。
- **滤镜处理：** 对 `<video>` 标签应用 CSS 滤镜（如 `sepia`, `contrast`, `noise` 叠加），使其融入羊皮纸背景。
- **混音通道：** 初始化 Howler.js。通道 1 循环播放环境底噪（雨声/风声）；通道 2 响应手势事件，触发瞬态音效（纸张摩擦、能量嗡鸣）。

### 3.6 结果封装与导出模块 (Export)

- **DOM 镜像生成：** 在后台隐式渲染一个不受屏幕视口限制的“完美版图文海报” DOM 节点。
- **图形渲染：** 调用 `html2canvas` 避开视频流（跨域或安全限制），仅将羊皮纸背景、卡牌图片、排版好的解牌文字渲染为 Canvas。
- **水印与防伪：** 在 Canvas 渲染前，在右下角动态注入当前时间戳（复古格式化）和“火漆印章” UI。导出为高质量 PNG 供用户下载。

## 4. 关键交互状态流转 (State Machine Diagram)

系统生命周期由全局状态机严格控制，主要状态枚举如下：

1. `INIT`: 资源预加载（图片、音频、模型）
2. `AWAITING_CONTRACT`: 等待用户同意“灵界链接”契约
3. `SETTING_INTENT`: 用户输入问题并选择牌阵
4. `SHUFFLING`: 洗牌模式（监测画圈手势） $\rightarrow$ 满5圈自动跃迁至下一状态
5. `DRAWING`: 抽牌模式（等待捏合手势选满指定数量的牌）
6. `REVEALING`: 卡牌翻转展示，并请求 AI/本地解析
7. `INTERPRETED`: 完整展示解读结果，激活导出功能
8. `SEALED_AND_RESET`: 导出完成，等待用户点击“再次凝神”重置状态

## 5. 非功能性需求 (NFR - Non-Functional Requirements)

- **性能保障 (Performance)：**
  - 首屏加载必须小于 3 秒。塔罗牌高清图需采用 WebP 格式并懒加载（首屏仅加载卡背）。
  - MediaPipe 手势识别帧率需保持在 30 FPS 以上，确保光标跟手无严重延迟。
- **隐私与安全 (Privacy & Security)：**
  - 严格声明：所有摄像头视频流均在浏览器本地内存处理，**绝对禁止**任何形式的图像数据网络传输。
  - 用户的 API Key 必须使用前端加密机制（如 base64 或简单的混淆）存入 LocalStorage，防范基础的 XSS 提取。
- **兼容性与降级 (Compatibility & Graceful Degradation)：**
  - 需兼容主流现代浏览器（Chrome, Safari, Edge）。
  - 在性能极低的设备上，自动关闭复杂的 CSS 阴影与粒子特效，保底提供基础的图文交互。

## 6. 演进路径 (Evolution Path)

- **Phase 1 (MVP - 本次设计边界)：** 完成基于 2.5D CSS 的所有核心流程，跑通 MediaPipe 双轨交互与学术派 AI 解读，验证产品核心概念。
- **Phase 2 (3D Upgrade - 未来规划)：** 引入 Three.js 替换现有的 2.5D 舞台；增加真实物理碰撞（Cannon.js）；将 MediaPipe 的 2D 坐标通过射线投射（Raycasting）升级为真正的三维空间抓取。
