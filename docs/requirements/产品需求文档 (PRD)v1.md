

# 产品需求文档 (PRD)

## 1. 文档概述 (Document Control)

- **项目名称：** 灵视塔罗 (Visionary Tarot) - 沉浸式互动占卜 Web App
- **文档版本：** V 1.0.0 (Phase 1: MVP)
- **产品形态：** SPA (单页面 Web 应用)，适配桌面端 Web
- **核心定位：** 结合 CV（计算机视觉）手势捕捉与大语言模型的强沉浸感、高仪式感神秘学体验平台。

## 2. 背景与目标 (Background & Objectives)

传统线上占卜多为枯燥的点击交互，缺乏仪式感。本项目旨在通过浏览器端轻量级手势识别（MediaPipe）与 2.5D 视效（CSS3D），模拟真实的物理占卜过程。MVP 阶段的核心目标是**验证“手势交互+学术派AI解读”双轨模式的市场接受度**，并在控制开发成本的前提下，打造出具有极高分享传播价值的闭环体验。

## 3. 术语表 (Glossary)

- **灵界链接 (Spirit Link)：** 本产品中对“请求并开启摄像头权限”的包装术语。
- **法器切换 (Focus Switch)：** 核心交互机制，指在“手势捕捉（灵气光球）”与“实体触控（鼠标/触控板）”之间的无缝降级与切换。
- **命运卷轴 (Destiny Scroll)：** 占卜结束后的结果展示页，以及最终生成的带火漆印章长图。
- **学术派解读 (Academic Interpretation)：** AI 生成结果的特定 Prompt 约束风格，包含符号学、元素对应及心理学原型解析。

------

## 4. 全局交互与状态说明 (Global Interaction)

本产品采用纯前端流转，不设立后端数据库保存用户数据。核心状态机拆分为四个主阶段：**入场 -> 洗牌 -> 抽牌 -> 解读**。

### 4.1 双轨交互引擎 (Dual-Track Engine)

- **主轨（手势识别）：** 开启摄像头后，隐藏系统鼠标指针，渲染一个发光的“灵气光球”跟随用户食指指尖。
- **副轨（鼠标降级）：** * 触发条件：用户拒绝摄像头权限、摄像头加载失败，或连续 10 秒未检测到有效手势。
  - 降级表现：界面角落“法器”图标暗下，恢复系统鼠标指针，用户可通过点击/拖拽完成相同操作。

## 5. 详细功能需求 (Functional Requirements)

### 5.1 模块一：入场与灵界链接 (Onboarding & Setup)

**需求描述：** 用户进入网站后的初始化设置及权限获取。

- **UI/UX 表现：** 动态复古羊皮纸背景，伴随低频环境白噪音（森林雨声/水晶钵）。
- **功能点：**
  1. **意图输入：** 提供多行文本框供用户输入心中的问题。
  2. **牌阵选择：** 下拉或平铺选择器（支持：单张启示、圣三角、六芒星、凯尔特十字）。
  3. **AI 秘钥配置 (隐藏入口)：** 界面边缘提供齿轮图标，展开后可输入并保存大模型 API Key（存入 LocalStorage）。
  4. **预授权仪式弹窗：** 点击“开始连接”后，不直接调用系统权限，而是弹出“灵界契约”。
     - 文案说明需明确：“仪式需要捕捉手势，但画面仅在本地运行，绝不上传。”
     - 确认契约后，方可触发浏览器原生 Camera 权限请求。

### 5.2 模块二：感知与洗牌 (Shuffling Mechanics)

**需求描述：** 沉浸式的洗牌过程。

- **UI/UX 表现：** 界面右下角显示添加了“复古/素描滤镜”的摄像头小窗（降低违和感）。屏幕中央展示 78 张叠加的牌堆。
- **手势动作：** 顺时针或逆时针“画圈”。
- **业务逻辑：**
  1. 系统监测手指移动轨迹的弧度累加。
  2. 背景浮现金色符文，随画圈动作转动并提亮，伴随纸张摩擦音效。
  3. **自动结束机制：** 当系统判定用户完成 **5 圈** 动作后，自动触发“牌堆整理对齐”动画，结束洗牌阶段。

### 5.3 模块三：抽取与翻转 (Drawing & Revealing)

**需求描述：** 用户根据所选牌阵抽取对应数量的卡牌。

- **UI/UX 表现：** 牌堆呈扇形或弧线展开（基于 2.5D CSS 透视）。
- **业务逻辑：**
  1. **光标悬停 (Hover)：** 灵气光球停留在某张牌上时，该牌需有 Y 轴向上的微小浮动反馈。
  2. **手势触发 (Draw)：** 识别大拇指与食指的“捏合”动作。判定成功后，卡牌伴随“灵能嗡鸣”音效飞入当前牌阵的待定槽位。
  3. **正逆位生成：** 抽牌入槽的瞬间，系统通过 `Math.random()` 决定该牌的正逆位状态（0 或 180 度），此时牌背依然朝上。
  4. **翻牌触发 (Reveal)：** 牌阵抽取完毕后，用户做“挥手”或再次“捏合”动作，卡牌依次沿 Y 轴 180° 翻转，展示牌面。

### 5.4 模块四：学术派解读 (Interpretation Engine)

**需求描述：** 根据用户是否有 API Key 决定解牌引擎，并展示结果。

- **无 API Key (默认降级模式)：** 调取本地 `meanings.json` 库，根据“牌名+正逆位”输出固定的短文本描述。
- **有 API Key (AI 驱动模式)：**
  - **输入组装：** 将 [用户问题] + [牌阵名称] + [抽取的牌名及正逆位] 组合注入预设 System Prompt。
  - **输出约束 (风格定调)：** AI 必须以“专业神秘学者”的口吻输出，包含四个固定维度：
    1. 【元素与星象对应】（如火元素、火星特质）
    2. 【符号学解析】（如对牌面皇冠、权杖的隐喻分析）
    3. 【深层意涵】（结合用户问题的理性学术剖析）
    4. 【启示建议】（转化为易懂的行为指引）
  - **展示效果：** 打字机流式输出（Streaming），配合卷轴向下展开的动画。

### 5.5 模块五：封印与导出 (Export & Sharing)

**需求描述：** 生成高传播价值的图片。

- **功能触发：** 页面底部提供“封印并带走这份启示”按钮。
- **图像合成规则 (基于 html2canvas)：**
  1. 渲染完整的羊皮纸背景框。
  2. 居中排版所抽卡牌（保持牌阵阵型与正逆位）。
  3. 附带完整的学术派解读文本。
  4. **印章与防伪：** 在图片右下角生成红色火漆印章图案，并附带时间戳（如：“公元二零二六年，星月交辉之夜”）。
- **下载行为：** 自动将 Canvas 转为 Base64 图片并触发本地下载。

## 6. 非功能性需求 (Non-Functional Requirements)

| **维度**                   | **要求指标**     | **说明**                                                     |
| -------------------------- | ---------------- | ------------------------------------------------------------ |
| **性能 (Performance)**     | 手势延迟 < 50ms  | MediaPipe 必须在 Web Worker 中运行，避免阻塞主线程 UI 渲染。 |
| **兼容性 (Compatibility)** | 主流桌面端浏览器 | Chrome (推荐), Edge, Firefox, Safari。需支持 WebGL 及 Web Audio API。 |
| **隐私安全 (Privacy)**     | 零图像数据上传   | 摄像头数据流 (MediaStream) 必须且只能在本地内存中处理，销毁后不留存。API Key 仅存放于浏览器 LocalStorage。 |
| **资源加载 (Loading)**     | 核心包 < 5MB     | 78 张高清塔罗牌必须进行 WebP 格式压缩；音频文件采用懒加载策略。 |

## 7. 数据与日志 (Data & Tracking - 内部评估用)

考虑到项目隐私性，不记录业务数据，仅在前端埋点统计以下匿名事件以优化转化率：

- `camera_auth_success` / `camera_auth_fail` (权限获取通过率)
- `gesture_fallback_triggered` (手势降级触发率，用于评估算法灵敏度)
- `result_downloaded` (生成图片下载量，衡量核心闭环完成度)
